# Memoring 企画書

この文書は、Memoring（メモリング）がなぜ必要で、誰にどんな価値をもたらし、どこへ向かうのかを説明する企画書である。投資家・協力者・開発者、そして Memoring に関わる AI エージェントが、製品の狙いと世界観を最短で理解できることを目的とする。データスキーマ・不変条件・CLI の詳細・要件定義はここでは扱わない。技術は「価値の根拠」として必要な範囲だけ触れ、それ以上は設計書に委ねる。

---

## 1. 一行で言うと

**Memoring: Own your AI memory.**
AI ツールに散らばる履歴を、あなたの記憶資産に変える。

Memoring は、Codex、Claude Code、ChatGPT、Claude、Gemini などの AI ツールがローカルに溜める会話・指示・応答・ツール実行・コマンド結果・ファイル差分・判断・制約・好み・作業パターンを取り込み、ユーザーが実効支配できる記憶資産として、自動で蓄積・整理・分類・抽象化・定着させ、必要なときだけ安全な文脈として取り出せるようにする OSS の **Sovereign Memory Loop（主権記憶循環）** である。

ここでいう own は包括的な法的所有権ではない。意味は user-controlled、すなわち「あなたのコピーをあなたが持ち、鍵・削除・持ち運び・出力をあなたが支配する」である。第三者コンテンツの法的所有権を主張するものではない。

---

## 2. 解決する課題

AI はすでに、あなたのことを覚え始めている。やり取りを重ねるほど、過去の決定や好み、守るべき制約を踏まえた応答が返ってくる。

しかし、その記憶はサービスごとに分断され、ローカルに散らばっている。Codex は Codex の、Claude Code は Claude Code の、ChatGPT は ChatGPT の中にしか記憶を持たない。あなたが別のツールに乗り換えれば、これまで積み上げた文脈はゼロからやり直しになる。さらにその実体は、各ツールがホーム配下などの隠しフォルダに溜めるセッションや履歴として、あなたの手元に確かに存在しているのに、横断して使える資産にはなっていない。

結果として、次のことが起きている。

- 同じ前提・同じ好み・同じ制約を、ツールを変えるたびに何度も説明し直している。
- 過去の判断や作業パターンが、どこかのローカルファイルに眠ったまま再利用されない。
- 記憶がベンダーのサービスに囲い込まれ、ユーザー自身が支配できない。

AI が日常になればなるほど、この分断のコストは積み上がる。Memoring はこの「分断され、散らばり、囲い込まれた記憶」を、ユーザーの手元で一つの記憶資産へ束ね直す。

## 3. Memoring とは何か / 何でないか

### 3.1 ログ保存ツールではない

ログを保存するだけならデータベースでよい。Memoring はデータベースではない。DB、object store、index は土台にすぎない。本体の価値は、履歴を「使える記憶と文脈」へ育て続けるループにある。

Memoring の価値は、次の連鎖が回り続けることにある。

```text
取得する → 蓄積する → 整理する → 分類する → 抽象化する → 定着する → さらに蓄積する
```

この連鎖が回ることで、散らばった履歴は単なる記録ではなく、再利用できる記憶と文脈になる。出口で生まれた文脈は次の AI 作業で使われ、その作業履歴がふたたび入口から取り込まれる。これで全体が閉じる。これが **Sovereign Memory Loop** であり、Memoring の本体である。

### 3.2 取得・自動ループ・出口

Memoring の働きは、取り込む・代謝する・手渡すの三拍子で捉えられる。

- **取得（Input）**: AI ツールのローカル蓄積を見つけて取り込む。入口は何も判断しない。まず壊さず取り込み、暗号化して蓄積する。
- **自動ループ（Loop）**: 取り込んだデータを整理・分類・抽象化し、長期記憶として定着（consolidate）させる。全自動で回り、手動の承認キューを挟まない。
- **出口（Output）**: 必要なときだけ、安全な文脈として取り出す。既定の出口は `.memoring/context.md` というファイルで、ほぼすべての AI ツールに読ませられる。

取り込みは「考えない」、整理と分類は「AI が考える」。この役割分担が Memoring の設計の中心にある。

## 4. なぜ必要か

Memoring が掲げる原則は、そのまま「なぜ必要か」の答えになる。

### 4.1 主権 — own your AI memory

記憶はベンダーのものではなく、ユーザーのものであるべきだ。Memoring の own は user-controlled を意味する。あなたのコピー、あなたの鍵、あなたの削除、あなたの持ち運び。記憶資産の支配権をユーザーの手に取り戻す。

### 4.2 local-first

記憶の置き場所はまずローカルである。クラウドに預けることが前提ではない。データはユーザーの端末で暗号化されて保持され、平文の生データがディスクに残ることはない。クラウドへ運びたければ運べるが、その場合もユーザー側で暗号化してから出す。復号鍵は常にユーザーの手にある。

### 4.3 model-independent

Memoring は特定の AI モデルやサービスに縛られない。Codex でも Claude Code でも ChatGPT でも、ローカルに履歴を溜めるツールであれば取り込める。記憶を使う AI も、ローカルモデル・主要プロバイダ API・コーディングエージェントのいずれにも開く。あるツールに最適化された記憶が、別のツールに乗り換えた瞬間に無価値になる、という囲い込みを起こさない。

### 4.4 OSS

Memoring はオープンソースである。記憶の主権をうたう以上、その仕組み自体が検証可能で、ユーザーの手元で完結し、特定企業のブラックボックスに依存しないことが筋である。

## 5. 誰にとって価値があるか

### 5.1 対象ユーザー

- AI コーディングエージェント / AI チャットを日常的に使っている個人。
- Claude Code / Codex のローカル履歴を資産化したいユーザー。
- 自分の AI 作業履歴を、将来の RAG / Context / Dataset に育てたいユーザー。

Memoring はまず single-user / local-first に絞って価値を立てる。チームや組織の中央管理は当面の対象ではなく、まず「個人が自分の AI 記憶を支配する」体験を作りきる。

### 5.2 ユースケース

- **AI 履歴の資産化**: これまで使い捨てられていた会話・判断・作業パターンを、横断して再利用できる記憶へ変える。
- **文脈の持ち越し**: 新しいセッションを始めるたびに、過去の決定・好み・制約を `context.md` として持ち越す。同じ前提を毎回説明し直す必要がなくなる。
- **将来の RAG / Context / Dataset**: 蓄積した記憶資産を、検索・文脈生成の基盤として育て、さらにその先で（ユーザーの同意と来歴管理のもとで）データセットへ展開する余地を残す。

最初に体験すべき価値はシンプルだ。新しい Claude Code / Codex のセッションを始めると、Memoring が過去の決定・好み・制約を `context.md` として持ち越してくれる。主役は検索ではなく、この「持ち越し」である。

## 6. 世界観 — 秩序を作り続けるループ

Memoring の設計は、いくつかの強い思想の上に立っている。読み物として通すために、その核心だけを述べる。

### 6.1 散逸構造 — 止めれば無秩序が勝つ

メモリとコンテキストは、放置すれば秩序から無秩序へ向かう。物事がひとりでに進む向きは、ふたつの傾きのニコイチで決まる。散らかろうとする傾き（エントロピー）と、エネルギーを放出して低く楽な状態へ落ち着こうとする傾き（エンタルピー）である。記憶も同じで、放っておけば中身は散らかり、ループも手を抜いて楽な近似へ流れ落ちる。秩序ある記憶は、この二つに逆らった状態であり、放置では保てない。

Memoring はこれに対し「閉じた系として秩序を守ろうとする」のではなく、「開いた系として秩序を作り続ける」**散逸構造**で応える。冷蔵庫が庫内を冷やすために電力を使い続け、その分の熱を外へ吐き出すように、局所の秩序を保つには、エネルギーを投入し続けて生じた無秩序を外へ排出し続けるしかない。ループそのものがその仕事であり、止めれば無秩序が勝つ。生命がエネルギーを使い続けて局所の秩序を保つのと同じ構造を、記憶に対して持たせる。

この原理から、設計を貫く一本の指針 **Metabolic Razor** が導かれる。

```text
秩序は構造とループで製造し、不可避な無秩序は隔離して排出する。
ユーザー依存の判断は自動化せず、判断しやすく差し出す（surfacing）に留める。
```

システムはユーザーの判断を肩代わりできないが、判断のための活性化エネルギーは下げられる。だから Memoring は、曖昧なものを無理に消し込もうとせず、しかし放置もしない。必要な判断はユーザーに見える形で差し出す。

### 6.2 The Undiluted is Truth — 原本は真実、Claim は主張

分類・要約・抽象化は必ず揺れる。モデルやルールが変われば結果も変わる。だからこそ Memoring は、解釈前の原本（Undiluted）を不変の真実として保ち、そこから作った知識は「根拠付きの主張（Claim）」として扱う。

- 原本は、削除や伏字以外では改変しない。
- 派生データは、いつでも作り直せる。
- Claim は不変の真実ではなく、根拠から再検証・再生成できる主張として扱う。

これは目的ではなく、ループを壊さないための安全原則である。最初の AI 出力だけを保存して原本を捨ててしまえば、二度とより良い記憶へ作り直せない。原本を残すからこそ、モデルが進歩するたびに記憶は育ち直せる。

### 6.3 ループは自分の誤差を増幅しない

秩序を作るはずのループ自身も、雑に動けば無秩序の源になりうる。分類や定着が甘ければ、誤った Claim を書き込んで無秩序を増やす。Memoring はこれを構造的に防ぐ。自分が生成した文脈を、自分の記憶の根拠（証拠）として食べ直さない。AI が言い換えただけの内容を独立した証拠に数えない。これは冷蔵庫が外へ吐き出した熱を吸い戻さないのと同じで、いったん系の外へ出した無秩序を入力として再摂取しない弁にあたる。こうした自己摂取の禁止があるからこそ、ループは自分の出力を再摂取して誤差を膨らませることなく、記憶の純増として働ける。「ループで閉じる」ことが正しいことの肝は、ここにある。

ここまでをひとつのエネルギーの物語として言い直すと、ループの仕事は、放っておけば乱れていく raw（低い秩序）を、実際に使える context（使える力）へ変換し、その途中で生じるムダな損失（散逸）を最小化することである。たとえとして、日常語の式で言えばこうなる（物理の主張ではない）。

```text
使える力 ＝ 持っている力 − 乱れ − ムダ漏れ
```

「持っている力」は取り込んだ raw のすべて、「乱れ」は放置で散らかる分、「ムダ漏れ」は無駄な再処理・重複保存・誤分類・不要な文脈注入といった散逸である。すでに述べた判断は、この一本の物語で読み直せる。idle で差分ゼロに収束するのは、変化のない所へ力を浪費しない散逸最小化であり、Ouroboros が自分の出力を食べ直さないのは、いったん外へ出した無秩序を再吸収しないことである。Metabolic Razor は、限られた力（秩序を作る仕事）を、秩序が作れて重要な所だけに投じる。The Undiluted is Truth は乱れの逃がし先（原本という排出先）を確保する。secret を event 単位で伏せるのは、安全のために受け入れる散逸である。ループが回り続ける限り、使える力は乱れとムダ漏れを差し引いてなお純増し続ける。

## 7. 提供価値の核

v0 が作る価値は 4 つに集約される。この 4 つだけで価値が成立するように設計する。とくに 1 と 3（取得と自動ループ）が Memoring の本体である。

1. **取得**: AI ツールのローカル蓄積から履歴を取り込む。
2. **蓄積**: 原本を壊さず、暗号化して保存する。
3. **ループ**: 整理・分類・抽象化・定着を自動で回す。
4. **出口**: `.memoring/context.md` を生成し、安全な文脈として手渡す。

機能の数で勝負するのではなく、「履歴を取り込み、自動で記憶化し、安全に持ち越す」という一本のループに価値を集中させる。

## 8. 差別化とポジショニング

Memoring の独自性は、記憶ツールとして当たり前に見える前提をいくつも外している点にある。

### 8.1 事前定義カテゴリを持たない

Memoring は、personal / private / social / work のような固定カテゴリをあらかじめ決めない。分類は AI が、蓄積されたデータに合わせて割り当てる。

理由ははっきりしている。事前にカテゴリを定義すると、それに当てはまらないデータが必ず出てくる。そのたびに例外対応と新ルール定義が増え、イタチごっこになる。Memoring は構造とループで全体を閉じ込め、不規則さは AI に処理させる。このほうが構造的に強い。

### 8.2 全自動 consolidate — 承認キューを持たない

多くの記憶系は「重要そうな記憶を 1 件ずつ人が承認する」設計を取る。Memoring はそれをしない。AI が候補を作り、Memoring が検証し、ルールと証拠を満たしたものだけが自動的に長期記憶へ定着する。ユーザーが逐一承認する代わりに、ユーザーは事後に統治する。気に入らない記憶を忘れさせ、訂正し、ピン留めし、封印する。

「AI は提案し、Memoring は検証し、ユーザーは事後に統治する」——この役割分担が、手動承認の摩擦なしに記憶を回し続けることを可能にする。

### 8.3 安全は「出力 Gate」で守る

Memoring の安全観は独特だ。「危ないものは記憶しない」のではなく、「記憶はするが、危ないものは出さない」。安全は記憶の入口ではなく、出力の門（Gate）で守る。

この門は、誰が読むか（Audience）と、どこまで出すか（Aperture）の 2 軸で判定する。鍵やトークンのような秘密はどの開放度でもそのまま出さない。未分類・判定不能なものは、判断できないなら出さない（fail-closed の Silence）。さらに、安全判定はランキング（並べ替え）より必ず前に来る。ランキングは品質調整であって、安全を緩める仕組みではない。

「記憶しても出さなければ守れる」という考え方は、原本を捨てずに保ちながら安全を担保するという Memoring の思想（原本は真実）と一貫している。

### 8.4 他の AI の記憶を取り込む「供給側」である

ChatGPT や Gemini が持つ「短期の context window vs 長期の memory」という枠組みは、一つのアシスタント内部の話である。Memoring はこの枠をそのまま採らない。

Memoring は記憶を使う側ではなく、**他の AI ツールの履歴を取り込んで記憶資産にする供給側**だからである。自前の会話バッファを持たず、各ツールがローカルに溜めた履歴を横断して束ねる。これは既存の AI 記憶機能と競合するのではなく、その上流に立つ独自のポジションである。どの AI が流行ろうと、その履歴を資産化する層として価値を持ち続ける。

## 9. 将来性とロードマップ

Memoring は v0 で核となるループを完成させ、そこから段階的に広げる。「いつかやる」ではなく、明確な線引きを持つ。

### 9.1 v0 が背負うもの

- Claude Code / Codex がローカルに溜める履歴の取り込み。
- 取り込んだ原本を壊さず暗号化して蓄積。
- 整理・分類・抽象化・定着を回す自動ループ。
- 出力 Gate で守られた `.memoring/context.md` の生成。
- 日本語を含む検索（exact match と n-gram fallback の常設）。

v0 は single-user / local-first / CLI + ローカル daemon に絞る。初期の Connector は Claude Code と Codex のローカルセッション、手動インポート、汎用 JSONL / Markdown の取り込みから始める。

### 9.2 v0.1 以降

- **ChatGPT / Claude / Gemini の export 取り込み**: 対応する AI ツールの幅を広げる。
- **local embedding / vector index**: 意味的な検索と、似たラベルの統合候補の提示を強化する。
- **MCP server polish**: 外部接続の標準的な受け皿を磨き込む。
- **span（行単位）追跡**: 文脈注入の追跡を、セッション単位からより細かい単位へ精緻化する。
- **alias citation**: 引用 ID をより扱いやすい形へ。
- **dataset builder**: 蓄積した記憶を、来歴と同意の管理のもとで学習用データセットへ展開する。

これらはロードマップ上の正当な線引きであり、v0 の核を肥大させないために意図的に後段へ置いている。

## 10. 市場性

AI コーディングエージェントと AI チャットの日常利用は、急速に裾野を広げている。使えば使うほど履歴が溜まり、その履歴が手元のローカルに散らばっていく構造は、ツールが普及するほど強まる。Memoring が解く「分断され、散らばり、囲い込まれた記憶」という痛みは、AI 利用が進むほど大きくなる。

ターゲットとなるのは、次の二つの流れが重なる層である。

- **AI を日常の道具として使い込む個人**: コーディングエージェントやチャットを毎日使い、過去の文脈を持ち越したいと感じている人々。
- **データ主権・プライバシーを重視する層**: 自分のデータを特定ベンダーに囲い込まれたくない、ローカルで支配したいと考える人々。

Memoring は local-first / OSS であることで、この二つの価値観に同時に応える。記憶の囲い込みに対する代替として、ユーザーの手元に主権を取り戻す選択肢を提供する。

---

## 11. まとめ

Memoring の核は、巨大な機能群ではない。

```text
AI ツールはローカルに履歴を溜める。
Memoring はそれを取り込み、自動ループで回して、
ユーザーが支配できる記憶と文脈に変える。
```

- 製品は、取得 → 蓄積 → 自動ループ → recall。DB は土台。
- 分類は事前定義せず、データに合わせて AI が行う。
- Claim は全自動で定着し、承認キューを持たない。
- 安全は出力 Gate で守り、ランキングは安全を緩めない。
- 記憶はユーザーが支配する。あなたのコピー、あなたの鍵、あなたの削除、あなたの持ち運び。

AI が散らばらせる記憶を、ユーザーの手元で一つの資産へ束ね直し、回し続ける。それが Memoring の Sovereign Memory Loop である。
